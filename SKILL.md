# Session 交接

使用简短的项目内交接文件，在不同 Codex session 之间传递工作状态。

## 操作缩写

此 skill 支持两个简短操作名称：

- `sho-summary`：总结当前 session，并创建一份新的交接文件。
- `sho-read`：读取最新或用户指定的交接文件，核实实际状态，汇报理解并等待用户确认。

推荐使用方式：

```text
$session-handoff sho-summary
$session-handoff sho-read
```

也可以使用自然语言：

```text
sho-summary
总结当前 session，准备交接
sho-read
读取交接文件，汇报理解并等待确认
```

如果用户只输入 `$session-handoff`，根据上下文判断模式：

- 用户准备开启新 session 时，执行 `sho-summary`。
- 用户希望恢复之前的任务时，执行 `sho-read`。
- 无法可靠判断时，简短询问用户。

## 交接文件位置和命名

将交接内容写入当前项目根目录：

```text
.codex/handoff/YYYY-MM-DD-HHmm-session-title.md
```

例如：

```text
.codex/handoff/2026-06-01-1530-session-handoff-skill-design.md
```

命名规则：

- 时间使用当前时区，格式为 `YYYY-MM-DD-HHmm`。
- 标题概括本次 session 的核心任务。
- 标题使用简短的小写英文或拼音，以连字符连接。
- 标题控制在 3-8 个词。
- 避免空格、中文、特殊字符和无意义的通用标题。
- 每次执行 `sho-summary` 时创建一份新的交接文件，不覆盖历史文件。

如果目录不存在，则创建 `.codex/handoff/`。

## 确认项目根目录

执行 `sho-summary` 或 `sho-read` 时：

1. 如果当前目录属于 Git 仓库，使用 `git rev-parse --show-toplevel` 返回的目录。
2. 如果当前目录不属于 Git 仓库，使用当前工作目录。
3. 写入交接文件前，告诉用户最终使用的项目根目录。
4. 不要将交接文件写入 skill 的安装目录。

## 创建交接文件

执行 `sho-summary` 时，只保留下一个 session 正确行动所需的信息。

目标长度为 600-1200 字。复杂任务可以适当增加，但应优先保持简洁。

使用以下结构：

```markdown
# Session 交接

更新时间：YYYY-MM-DD HH:mm
项目根目录：/绝对路径
Session 标题：简短标题

## 本次 Session 摘要
简要说明本次 session 讨论了什么、处理了什么问题，以及工作如何推进到当前状态。
使用 3-8 条要点。

## 最终目标
用户最终希望完成什么。

## 当前状态
已经完成哪些工作。
当前哪些内容可以正常运行。
哪些事项仍然尚未完成。

## 下一步行动
1. 下一个 session 首先应该执行的具体动作。
2. 按优先级排列的其他待办事项。

## 已确认的决策
- 已经确定的重要选择，以及作出选择的原因。
- 必须保留的限制条件或用户偏好。

## 相关文件
- `path/to/file`：发生了什么改动，或者为什么值得关注。

## 验证情况
- 已经运行的检查及其结果。
- 尚未执行的检查。

## 阻塞问题
- 尚未解决的报错、缺失的信息或风险。
- 如果没有阻塞问题，填写“无”。

## 新 Session 启动提示词
执行 `sho-read`：读取 `.codex/handoff/` 中最新的交接文件，
检查当前文件和 git diff，说明对已有工作的理解、建议的下一步行动和风险，
然后询问用户是否继续。获得用户明确授权前，不要修改文件或执行后续任务。
```

## 写入前检查

执行 `sho-summary` 时：

1. 确认当前项目根目录。
2. 如果项目使用 Git，运行 `git status --short`。
3. 如果修改过代码或文档，检查相关 diff。
4. 明确区分已经验证的事实和仍未验证的推测。
5. 记录尚未完成的用户请求，包括对话后期新增的要求。
6. 只有当命令的准确形式会影响后续工作时，才记录完整命令。
7. 如果已经存在交接文件，仍然创建一份新的文件，不覆盖历史记录。

## 写入后反馈

完成 `sho-summary` 后：

1. 告诉用户交接文件的保存位置。
2. 使用 1-3 条要点简要说明本次摘要记录了什么。
3. 简短说明下一个 session 首先应该做什么。
4. 将以下提示词交给用户，用于开启新的 session：

```text
执行 sho-read：读取 `.codex/handoff/` 中最新的交接文件，
检查当前文件和 git diff，说明对已有工作的理解、建议的下一步行动和风险，
然后询问我是否继续。获得我的明确授权前，不要修改文件或执行后续任务。
```

## 读取交接文件

执行 `sho-read` 时：

1. 确认项目根目录。
2. 列出 `.codex/handoff/` 中的交接文件。
3. 如果用户指定了交接文件，则读取该文件。
4. 如果用户没有指定，则读取文件名时间最新的交接文件。
5. 如果项目使用 Git，检查 `git status --short`。
6. 对照当前文件，核实交接内容是否仍然准确。
7. 将交接文件视为工作摘要，而不是绝对事实。
8. 向用户简要说明对已有工作的理解。
9. 列出建议的下一步行动，以及需要用户注意的风险或歧义。
10. 询问用户是否继续执行。
11. 获得用户明确授权前，不要修改文件或执行后续任务。

## 压缩规则

应该保留：

- 本次 session 讨论和推进的内容
- 用户的最终目标
- 已完成的工作
- 剩余任务
- 重要决策和限制条件
- 相关文件路径
- 验证结果
- 报错和阻塞问题
- 明确的下一步行动

应该省略：

- 完整聊天记录
- 重复讨论
- 已放弃方案的探索过程
- 通用背景知识
- 文件中已经存在的大段代码
- 尚未要求实现的未来改进
- 与后续行动无关的细节

优先引用文件路径和行号，例如 `src/app.ts:42`，不要粘贴大段代码。

## 安全规则

- 创建交接文件时，不要修改源代码。
- 没有实际运行的检查，不要声称已经通过。
- 不要撤销与当前任务无关的用户改动。
- 读取交接文件后，以当前文件系统状态为准。
